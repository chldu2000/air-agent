# v0.3 Skills Tracing 补强设计

## 背景

v0.3 已通过 `RunEvent`、`EventDispatcher` 和结构化日志覆盖 LLM 调用、工具调用、重试、错误与完成状态。然而，Skills 的路由与注入仍是运行轨迹中的盲区：

- 无法确认路由器评估了哪些候选 Skill。
- 无法查看路由器原始返回以及最终识别出的 Skill。
- 路由失败会静默降级为空列表，无法从运行事件判断失败原因。
- 无法确认哪些 Skill 内容实际注入了本轮上下文。
- streaming 与非 streaming 路径分别实现 Skill 路由和注入，难以保证观测语义一致。

本补强以运行调试为主要目标，回答“本轮为什么选择或没有选择某个 Skill”。它不改变现有 Skill 选择逻辑，也不要求模型额外生成选择理由。

## 目标

- 复用 v0.3 的 `RunEvent` 与 `EventDispatcher`，补齐 Skill 路由和注入事件。
- 完整记录路由器原始返回，便于定位匹配、解析和命名问题。
- 让路由失败显式可见，同时保持降级为空 Skill 的兼容行为。
- 统一 streaming 与非 streaming 模式的 Skill 路由、注入和事件语义。
- 为 v0.4 Provider 抽象保留 Provider 无关的路由结果接口。

## 非目标

- 不改变现有 Skill 路由 prompt 或匹配规则。
- 不要求模型解释选择理由，也不记录逐个候选 Skill 的推理过程。
- 不增加 Skills 专用 tracing 开关；继续复用 `enable_tracing` 和 `log_events`。
- 不记录用户完整输入或 Skill 完整内容。
- 不增加聚合指标、可视化控制台或外部观测后端。
- 不改变 `Agent.run()`、`Response` 或 `StreamEvent` 的公共行为。

## 设计决策

### 结构化路由结果

新增 Provider 无关的 `SkillRouteResult`，由 Router 返回路由过程的稳定结果：

```python
@dataclass
class SkillRouteResult:
    matched_skills: list[Skill] = field(default_factory=list)
    raw_output: str = ""
    duration_ms: float | None = None
    unrecognized_names: list[str] = field(default_factory=list)
    error_type: str | None = None
    error_message: str | None = None
```

字段语义：

- `matched_skills`：按候选 Skill 原始顺序排列的有效命中项。
- `raw_output`：路由模型返回的完整文本；路由调用失败时为空字符串。
- `duration_ms`：路由调用和结果解析总耗时。
- `unrecognized_names`：路由器返回但候选列表中不存在的名称，按原始返回顺序去重。
- `error_type`、`error_message`：路由失败的稳定摘要；成功时均为 `None`。

结果对象不保存异常实例，避免异常对象泄漏到公共接口、日志或未来的序列化边界。

### Router 接口兼容

`SkillRouter` 保留现有抽象 `match()` 契约，并新增默认 `route()` 包装：

```python
@abstractmethod
async def match(self, user_input: str, skills: list[Skill]) -> list[Skill]:
    raise NotImplementedError
```

默认 `route()` 实现负责计时、调用 `match()`，并将成功或失败转换为 `SkillRouteResult`。这保证现有只实现 `match()` 的自定义 Router 继续可用，并自动获得耗时、命中项和错误摘要。由于旧 Router 不暴露底层模型输出，其 `raw_output` 为空字符串。

`LLMSkillRouter.route()` 负责：

- 执行路由调用并计时。
- 保存模型完整原始返回。
- 解析有效命中和无法识别的名称。
- 捕获异常并返回带错误摘要的 `SkillRouteResult`。
- 在无候选 Skill 时直接返回空成功结果，不调用模型。

`LLMSkillRouter.match()` 作为兼容入口，仅返回 `(await self.route(user_input, skills)).matched_skills`。

Router 不直接依赖 `EventDispatcher`、`RunEvent`、`run_id` 或 `conversation_id`。这使自定义 Router 可独立测试，也避免在 v0.4 Provider 迁移时把 tracing 逻辑绑定到具体模型客户端。

### Agent 统一路由与注入

`Agent` 新增私有方法 `_route_and_inject_skills()`，供 `_run()` 与 `_run_stream()` 共同调用。该方法接收当前消息历史与运行上下文，完成：

1. 若没有已加载 Skill，直接返回原历史且不产生 Skill 事件。
2. 发出 `skill_route_start`。
3. 调用 `SkillRouter.route()`。
4. 根据结果发出 `skill_route_end` 或 `skill_route_error`。
5. 将有效命中的 Skill 内容按当前行为注入历史。
6. 每成功执行一次 Skill 注入操作，发出一个 `skill_injected`。
7. 返回注入后的历史。

路由只在每次 `Agent.run()` 开始时执行一次，不随 ReAct iteration 重复执行。该行为与当前实现一致。

## 事件模型

本补强继续使用现有 `RunEvent` 字段，不新增事件专用数据类型。

### `skill_route_start`

在调用 Router 前发出：

```json
{
  "type": "skill_route_start",
  "run_id": "run_example",
  "conversation_id": "optional",
  "metadata": {
    "candidate_names": ["brainstorming", "debugging"],
    "candidate_count": 2,
    "router": "LLMSkillRouter"
  }
}
```

### `skill_route_end`

Router 成功返回时发出，包括返回 `none` 或没有有效命中的情况：

```json
{
  "type": "skill_route_end",
  "run_id": "run_example",
  "content": "brainstorming, unknown-skill",
  "duration_ms": 42.1,
  "metadata": {
    "matched_names": ["brainstorming"],
    "unrecognized_names": ["unknown-skill"]
  }
}
```

`content` 完整保存路由器原始返回。当前 Router prompt 只要求返回 Skill 名称，因此该字段通常较短；用户仍应将 tracing 输出视为可能包含模型生成内容的日志。

### `skill_route_error`

Router 调用或解析发生异常时发出：

```json
{
  "type": "skill_route_error",
  "run_id": "run_example",
  "content": "API unavailable",
  "duration_ms": 41.8,
  "metadata": {
    "error_type": "RuntimeError",
    "fallback": "no_skills"
  }
}
```

事件发出后，Agent 继续以没有匹配 Skill 的方式运行。Skills tracing 不应使原本可完成的运行失败。

### `skill_injected`

每个实际注入历史的 Skill 发出一次，顺序与注入操作顺序一致：

```json
{
  "type": "skill_injected",
  "run_id": "run_example",
  "name": "brainstorming",
  "metadata": {
    "path": "/path/to/brainstorming",
    "content_length": 318
  }
}
```

事件不包含 Skill 完整内容。`content_length` 使用 Python 字符串长度，用于确认是否加载了空内容或意外的大型 Skill。

## 事件序列

成功命中：

```text
skill_route_start
skill_route_end
skill_injected
llm_start
[existing LLM and tool events]
done
```

多个 Skill 命中时，在 `skill_route_end` 后按注入操作顺序产生多个 `skill_injected`。

无命中：

```text
skill_route_start
skill_route_end
llm_start
[existing LLM and tool events]
done
```

路由失败并降级：

```text
skill_route_start
skill_route_error
llm_start
[existing LLM and tool events]
done
```

streaming 与非 streaming 模式必须产生相同的 Skills 事件序列和字段语义。

## 错误处理与兼容性

- Router 无候选 Skill 时返回空成功结果；由于 Agent 不调用 Router，因此不产生 Skills 事件。
- Router 返回空字符串或 `none` 时视为成功无命中，产生 `skill_route_end`。
- Router 返回未知名称时视为成功，将名称写入 `unrecognized_names`，不产生错误事件。
- Router 调用或解析抛出异常时产生 `skill_route_error`，并降级为空命中。
- tracing handler 或结构化日志失败仍遵循 `EventDispatcher` 的 best-effort 行为，不影响 Agent 运行。
- tracing 未启用时，事件创建调用可以保留，但 `EventDispatcher` 不分发事件；Agent 返回值和路由行为保持不变。
- 现有调用 `SkillRouter.match()` 以及只实现 `match()` 的自定义 Router 保持可用。
- `SkillRouteResult` 从 `air_agent.skills` 和顶层 `air_agent` 导出，作为自定义 Router 的公共契约。

## 与 v0.4 Provider 抽象的关系

`SkillRouteResult` 不包含 OpenAI 响应对象，也不假设 Router 使用哪种 Provider。v0.4 将 `LLMSkillRouter` 从 OpenAI client 迁移到 `LLMProvider` 时，只需替换内部调用方式；Agent 的 Skills tracing 事件和结果解析接口保持不变。

v0.4 implementation plan 在执行时应将本设计视为 Router 迁移的前置公共接口，避免重新引入只返回 `list[Skill]` 的实现。

## 测试策略

### Router 单元测试

- `route()` 返回完整 `raw_output`、有效命中、未知名称和非空耗时。
- `none`、空输出和无候选 Skill 返回空成功结果。
- 重复名称不会产生重复未知名称或重复匹配项。
- 模型调用失败时返回 `error_type`、`error_message` 和耗时。
- `match()` 继续只返回有效命中列表。

### Agent 集成测试

- 成功命中产生 `skill_route_start`、`skill_route_end` 和 `skill_injected`。
- `skill_route_start` 包含完整候选名称和 Router 类型。
- `skill_route_end.content` 等于 Router 完整原始返回。
- 未知名称写入 `unrecognized_names`，但不被注入。
- 无命中时不产生 `skill_injected`。
- 路由失败产生 `skill_route_error`，Agent 仍正常完成。
- 多个 Skill 的 `skill_injected` 顺序与注入操作顺序一致。
- tracing 关闭时不分发 Skills 事件。
- streaming 与非 streaming 模式产生相同语义的 Skills 事件。

### 回归测试

- 当前 Skill 元数据与内容注入测试继续通过。
- 当前 LLM、工具、重试和完成事件顺序保持不变，只在启用 Skills 时新增前置 Skills 事件。
- 无 `skills_dir` 的运行事件序列不变。

## 验收标准

- 开发者能够从一次 tracing 记录中确定候选 Skill、Router 原始返回、有效命中、未知名称和实际注入项。
- 路由失败不再静默，但仍保持为空 Skill 的降级行为。
- streaming 与非 streaming 模式使用同一条路由与注入路径。
- 未启用 tracing 时，现有公共行为和返回类型保持兼容。
- 自定义 Router 可通过 `SkillRouteResult` 接入，而无需依赖 tracing 组件。
- 全量测试通过，README 中的 tracing 事件说明包含新增 Skills 事件。
